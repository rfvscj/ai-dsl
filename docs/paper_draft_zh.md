# TokenLite: 面向大语言模型代码生成的低 Token Python 风格 DSL

## 摘要

大语言模型已经成为代码生成、代码解释与代理式编程的重要接口，但当前主流编程语言的语法设计主要服务于人类开发者，而非面向模型的 token 效率。本文提出一种面向大语言模型代码交互场景的 Python 风格领域特定语言 TokenLite。该语言的核心目标不是提升人类可读性，而是在保持 Python 运行时兼容性的前提下，压缩高频代码模式在 tokenizer 下的表示成本。TokenLite 采用“DSL 源码生成 + 本地翻译为 Python 执行”的架构，将语句层压缩与表达式级宏相结合，以减少模型在代码生成与代码执行链路中的 token 消耗。我们实现了一个可运行原型，包括缩进敏感语法、宏展开器、DSL 到 Python 的翻译器，以及针对 OpenAI 兼容编码的 token 基准工具。实验表明，在一个代表性的小任务上，相对于直接生成 Python，TokenLite 可将代码输出 token 从 74 降至 54，降幅 27.03%，并将完整回答 token 从 127 降至 108，降幅 14.96%。结果说明，若 DSL 仅在关键字层面缩写，收益有限；但若对高频表达式结构进行专门压缩，则可以获得可测量的真实 token 优势。本文进一步讨论了该方向在 AI-first 编程接口、代理运行时、模型协作式软件工程中的潜力与限制。

**关键词**：DSL，大语言模型，代码生成，token 压缩，Python，编译器

## 1. 引言

当前代码生成模型广泛使用 Python、JavaScript、C++ 等通用语言作为输出媒介。然而，这些语言的历史设计目标是面向人类程序员，而不是面向基于子词切分的统计模型。对于大语言模型而言，代码并不只是抽象语法树，而是具体的 token 序列；模型的推理成本、上下文占用、调用费用以及多轮交互的容量都直接受到 token 数量影响。

这一事实引出一个自然问题：是否可以设计一种面向 AI 的代码表示，使其在保留语义表达能力的同时，比 Python 在模型 tokenizer 下更紧凑？本文认为答案是肯定的，但需要谨慎界定目标。我们的目标不是构造一个通用新语言，也不是追求对人类更加友好，而是探索一种 **AI-first 的代码表层表示**：它应当更适合模型生成、更适合模型读取，并可以无缝落地到既有 Python 生态。

本文提出 TokenLite，其设计原则如下：

1. 以 Python 作为后端运行时，不破坏现有执行生态。
2. 优先压缩高频模式，而不是盲目重造整门语言。
3. 优先以真实 tokenizer 度量收益，而不是只看字符数。
4. 允许牺牲部分人类可读性，换取模型交互效率。

我们实现了一个最小但完整的原型，并使用真实 tokenizer 对生成链路和执行链路进行分析。原型验证表明，仅做 Python 关键字缩写几乎不能带来显著 token 收益；只有当 DSL 直接压缩列表推导、filter-map 组合、聚合操作等高频表达式结构时，token 优势才会显现。

## 2. 问题定义

我们关注如下场景。用户向模型提出任务，模型需要：

1. 生成一段可执行代码；
2. 在后续对话中解释、执行、修订或组合这段代码；
3. 最终由本地运行时或外部系统执行实际逻辑。

在这一过程中，存在两类核心 token 成本：

1. **代码生成成本**：模型从任务描述生成代码时，输出代码本身所占用的 token。
2. **代码执行成本**：模型在解释器、代理或执行提示词中再次接收该段代码时，代码作为输入上下文所占用的 token。

如果一种 DSL 可以降低这两类成本，并且其输出能够被本地快速翻译为 Python，那么它就具备实际工程价值。

形式化地，给定任务描述 $T$，令模型输出代码为 $C$，执行提示模板为 $E(\cdot)$，tokenizer 为 $\tau(\cdot)$。我们关心以下量：

- 生成输入成本：$|\tau(T)|$
- 生成输出成本：$|\tau(C)|$
- 执行输入成本：$|\tau(E(C))|$

本文的目标是在语义等价的前提下，找到一种代码表示 $C_{dsl}$，使其相对于 Python 表示 $C_{py}$ 满足：

- $|\tau(C_{dsl})| < |\tau(C_{py})|$
- $|\tau(E(C_{dsl}))| < |\tau(E(C_{py}))|$

或者在部署路径上满足：

- 模型输出 $C_{dsl}$；
- 本地翻译器将其转换为 $compile(C_{dsl}) = C_{py}$；
- 由 Python 运行时执行。

## 3. 方法

### 3.1 总体架构

TokenLite 采用三层结构：

1. **DSL 表示层**：由模型直接输出，目标是压缩 token。
2. **翻译层**：将 DSL 翻译为 Python 源码。
3. **运行层**：直接复用 Python 解释器及其生态。

该架构的关键优势是将“面向模型的表示优化”和“面向执行的生态兼容”解耦。模型不必直接输出冗长 Python，而是输出更短的 DSL；系统侧再负责确定性地恢复成 Python。

从更长远的角度看，本文并不将 Python 视为唯一目标。多后端设计的意义在于：

1. **面向多语言项目集成**：现实软件项目可能分别以 Python、C++、Rust、Go 等语言为主体，因此同一 DSL 若能降到不同宿主语言，将更容易嵌入既有工程体系。
2. **面向不同执行形态**：除“编译到宿主语言后再运行”外，未来还可以探索直接降到更底层中间表示甚至独立二进制，以减少解释器依赖并追求更高执行效率。

进一步说，独立二进制路线并不只是工程优化选项，而可能是 AI-DSL 发展的一个终局方向。如果 AI-DSL 最终能够直接承载项目构建，那么编译器、运行时、构建系统、调试接口乃至部署链路都可以围绕 AI-DSL 自身建立。在这种形态下，AI-DSL 不再只是“帮助生成宿主语言源码的中间前端”，而会成为 AI 构建软件项目的核心表示。

### 3.2 语法设计

原型保留 Python 表达式的大部分能力，只压缩两类内容。

第一类是语句层结构：

- `f name(args)` 对应 `def name(args):`
- `? cond` 对应 `if cond:`
- `:` 对应 `else:`
- `r expr` 对应 `return expr`
- `= x expr` 对应赋值
- `p expr` 对应 `print(expr)`

第二类是表达式级压缩宏，用于覆盖高频 Python 模式：

- `F(seq, cond)`：过滤
- `M(seq, expr)`：映射
- `FM(seq, cond, expr)`：过滤与映射组合
- `S(seq)`：求和
- `A(seq, cond)`：任意满足
- `E(seq, cond)`：全部满足

其中 `_` 作为占位变量。举例而言：

```text
= squares FM(nums,_%2==0,_*_)
```

可编译为：

```python
squares = [it * it for it in nums if it % 2 == 0]
```

这里的设计重点不在于语法优雅，而在于把 Python 中一串高频而离散的 token 压缩成更短、更规则、更容易让模型复现的模式。

### 3.3 编译策略

现阶段原型更准确地说是轻量文本到文本翻译器：

1. 对缩进敏感行进行归一化；
2. 解析语句头；
3. 在表达式中扫描宏调用；
4. 展开为标准 Python 表达式；
5. 输出可直接执行的 Python 源码。

虽然该实现尚未构建正式 AST，但它足以支持快速试验语言表面设计与 token 统计。该选择符合原型阶段目标：以最低工程成本验证 DSL 是否能在真实 tokenizer 下获得收益。

### 3.4 Token 统计方法

本文不采用字符数作为主要指标，而直接使用 OpenAI 兼容编码：

- `o200k_base`
- `cl100k_base`

对于同一任务，我们比较三种场景：

1. 模型直接输出 Python；
2. 模型直接输出 DSL；
3. 模型输出 DSL，系统本地翻译为 Python 后再进入执行提示。

这样可以分别观察：

- DSL 对生成输出成本的影响；
- DSL 对执行输入成本的影响；
- “先 DSL 后编译”的系统路径是否仍然具有整体收益。

## 4. 原型实现

我们实现了一个完整原型系统，包含以下组件：

1. **翻译器/编译器原型**：支持缩进语法、宏识别、表达式展开与目标代码生成。
2. **命令行工具**：支持 `compile`、`run`、`stats` 等命令。
3. **基准脚本**：支持代码级与管线级 token 统计。
4. **样例程序**：用于验证语法正确性与收益。

原型的关键特征在于，它不是一个单纯概念说明，而是一套可运行、可复现实验的最小系统。用户可以直接编写 DSL 文件、翻译为 Python、执行结果，并在同一项目中比较 Python 与 DSL 的 token 差异。

## 5. 实验

### 5.1 任务

我们选取一个简单但具有代表性的程序任务：

“给定一组整数，保留偶数，将其平方后求和，并打印结果。”

对应 Python 实现为：

```python
def even_square_sum(nums):
    evens = [n for n in nums if n % 2 == 0]
    squares = [n * n for n in evens]
    return sum(squares)

data = [1, 2, 3, 4, 5, 6]
result = even_square_sum(data)
print(result)
```

对应 TokenLite 实现为：

```text
f even_square_sum(nums)
  = squares FM(nums,_%2==0,_*_)
  r S(squares)

= data [1, 2, 3, 4, 5, 6]
= result even_square_sum(data)
p result
```

### 5.2 结果

在 `o200k_base` 和 `cl100k_base` 两套编码上，结果一致。

#### 代码级比较

- Python 代码 token：74
- DSL 代码 token：54
- 节省 token：20
- 相对降幅：27.03%

#### 带任务上下文的回答比较

- Python 回答 token：127
- DSL 回答 token：108
- 节省 token：19
- 相对降幅：14.96%

#### 管线级比较

Python 方案：

- 生成输入 token：59
- 生成输出 token：74
- 生成总 token：133
- 执行输入 token：89

DSL 方案：

- 生成输入 token：59
- 生成输出 token：54
- 生成总 token：113
- 执行输入 token：70

DSL 编译后执行方案：

- 生成输入 token：59
- 生成输出 token：54
- 生成总 token：113
- 执行输入 token：75

### 5.3 分析

实验揭示了两个重要结论。

第一，仅在关键字层面对 Python 做缩写，收益非常有限。我们在原型早期曾观察到，简单将 `def/if/return/print` 等替换为单字符缩写后，真实 token 几乎不变。这是因为现代 tokenizer 已经对高频编程关键字做了有效压缩。

第二，真正有效的压缩来自 **高频结构模式** 的整体替代。例如 `FM(nums,_%2==0,_*_)` 将多个 Python 关键词、标识符和推导结构折叠成了一个规则宏，使得模型输出更短、输入上下文也更短。因此，AI-first DSL 的重点不应放在“人类语言式精简”，而应放在“面向 tokenizer 的高频模式编码”。

## 6. 讨论

### 6.1 与传统 DSL 的区别

传统 DSL 往往服务于特定业务领域，如 SQL、Verilog、正则表达式等，其目标是表达力或领域适配。TokenLite 的目标不同，它服务于 **模型交互成本优化**。因此，它不是围绕某个外部业务领域，而是围绕“语言模型如何最便宜地表达程序结构”来设计。

### 6.2 与普通代码压缩的区别

普通代码压缩往往关注字符数、文件体积或网络传输，而本文关注的是 tokenizer 之后的 token 长度。字符更短并不必然意味着 token 更少，因此不能直接借用传统 minify 思路。DSL 的所有设计都应在真实 tokenizer 上做闭环评估。

### 6.3 与 `caveman` 一类压缩表达项目的区别

在形成这一想法之后，我也进一步去查找了相近方向的工作，并看到了 `JuliusBrussee/caveman` 这类项目。它将 agent 输出压缩为更简短的“caveman-speak”，并加入了“文言文 mode”这类高信息密度表达实验。将其放在这里讨论，主要是为了说明两条路线在目标层面并不相同。

`caveman` 更接近一种 **输出风格压缩层**，即让模型“说得更短”；而本文关心的是一种 **可执行的 AI 编程中间表示**，以及围绕它建立端侧翻译器、运行路径和训练接口。换句话说，本文并不把“更短的说话风格”视为最终目标，而是希望：

1. 模型直接以 AI-DSL 作为主要代码表示进行生成；
2. 端侧由翻译器把 AI-DSL 快速恢复成 Python、C++ 或其他宿主形式；
3. 更长远地，模型训练本身也逐步转向这种高信息密度、可规则恢复的表示层。

如果把这个方向继续推远，那么问题就不再只是“怎样让模型说得更短”，而是：是否可以让模型直接使用一种更适合 AI 交换、规划与构建的语言层？本文倾向于认为答案可能是肯定的。更长远的目标不是停留在 skill 或提示词层面的压缩风格，而是走向“端侧翻译器 + AI language / AIDL-native 训练”的形态：模型直接在这种高信息密度表示层上学习、生成和协作，端侧系统再把它恢复到宿主语言或人类可读形式。

### 6.4 潜在应用

该方向在以下场景具有潜力：

1. **代理式编程**：模型频繁生成、修改、执行代码时，累积 token 成本可能显著下降。
2. **上下文受限系统**：在长对话和多轮补丁生成中，短代码表示有助于释放上下文容量。
3. **多模型协作**：一个模型用 DSL 生成计划，另一个系统在本地编译和执行。
4. **跨语言工程适配**：同一 DSL 可针对不同项目主语言落到不同后端，例如 Python 后端适配脚本和数据项目，C++ 后端适配性能敏感或原生系统项目。
5. **AI-native runtime**：未来可将 `tool`、`memory`、`plan`、`retry` 这类 AI 工作流原语纳入语言层。
6. **AI-DSL 原生项目形态**：长期看，项目本身可以直接以 AI-DSL 作为主要构建表示，AI 在统一的低 token 语义层上完成生成、修改、构建和执行。

### 6.5 局限性

当前原型存在明显局限：

1. 它仍然是 Python 的前端，而非独立语义系统。
2. 宏系统覆盖范围有限，只处理少量高频模式。
3. 编译器仍是文本级 lowering，错误定位与语义检查较弱。
4. 基准任务规模较小，尚未建立系统性评测集。
5. 目前只统计 token，不统计模型正确率、鲁棒性与训练可迁移性。

因此，本文结论应被理解为“方向可行、原型有效”，而非“最终方案已定型”。

## 7. 未来工作

未来可以从以下几个方向推进：

1. **构建 AST 编译器**：增强语法分析、错误报告和可组合变换能力。
2. **扩展多后端 lowering**：除 Python 外，逐步支持 C++、Rust 等后端，以验证 DSL 对不同宿主项目的适配价值。
3. **设计二进制执行路径**：在宿主语言后端之外，探索 `DSL -> IR -> binary` 的独立运行路线，以提升执行效率并减少运行时依赖。
4. **扩展压缩宏**：覆盖 dict 构造、group-by、reduce、异常处理、函数式管道等高频模式。
5. **引入 AI 原语**：加入 `tool`、`call`、`plan`、`memory`、`retry` 等执行结构。
6. **建立大规模评测集**：覆盖算法题、数据处理、代理任务、工具编排等多种工作负载。
7. **比较不同 tokenizer**：评估该方法对不同模型家族的泛化性。
8. **研究训练友好性**：分析模型对 DSL 学习成本、泛化能力与错误模式的变化。
9. **探索 AI 自己的沟通语言**：在传统编程语言之外，研究一种更贴近 AI 内部交换和规划的表示层，例如尝试中英混合、去除低价值修饰、强化规则词和结构词，并通过端侧快速规则式翻译器把该表示映射回面向人类或宿主语言的形式。

更激进的设想是把 AI-DSL 继续向“AI-native communication language”推进。一个可能可行的方向，并不是立刻发明完全脱离人类语言的新系统，而是从信息密度更高、token 更经济的人类语言材料出发，例如中文和英文的混合表达：保留结构词、操作词、约束词和少量必要符号，主动省去对 AI 推理帮助不大的修饰成分，再由一个规则式、近乎即时的翻译器把它恢复到宿主语言或面向人的说明文本。按照当前直觉，这条路线可以理解为“无损编码，可损解码”：AI 之间交换的编码表示尽量稳定、紧凑且可逆，而面向人的还原层则可以允许一定程度的重述与展开，只要求语义保持，不必逐 token 对齐。若这一方向成立，那么未来的 AI 编程接口甚至未必要建立在任何现有人类编程语言之上，而可能演化出一套专门服务于模型协作、规划和构建的软件表示系统。

## 8. 结论

本文提出了一个面向大语言模型代码交互场景的 Python 风格低 token DSL 原型 TokenLite。通过“模型输出 DSL，本地翻译为 Python 执行”的路径，我们证明了在真实 tokenizer 下压缩代码 token 是可行的。实验表明，若 DSL 仅压缩关键字，其收益几乎可以忽略；而当 DSL 面向高频表达式结构进行专门设计时，可以在生成与执行链路上取得实际收益。

这一结果说明，未来的 AI-first 编程接口不必完全沿用人类优先的通用语言表层形式。相反，可以通过专门为模型设计的中间表示，在不放弃现有运行时生态的前提下，降低交互成本并扩展系统设计空间。更长远地看，AI-DSL 甚至可能从“宿主语言之前的一层表示”演化为“AI 直接构建项目的原生表示”，并围绕其形成完整工具链。我们认为，这一方向值得进一步系统研究。
